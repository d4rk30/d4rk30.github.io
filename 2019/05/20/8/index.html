在Web页面中怎样使用对话框（Dialog）

发表于 2019-05-20 更新于 2019-07-19

一直疑惑该怎样使用对话框，在不断实践中，自己总结的一些规则。

0x00 什么是对话框

对话框英文叫：Dialog（在AntD的组件中叫：Model，后面解释一下）
先看一下维基百科对对话框解释：

图形用户界面中，对话框（又称对话方块）是一种特殊的视窗, 用来在用户界面中向用户显示信息，或者在需要的时候获得用户的输入响应。之所以称之为“对话框”是因为它们使计算机和用户之间构成了一个对话——或者是通知用户一些信息，或者是请求用户的输入，或者两者皆有。

通俗一点的解释，就是出现在页面上的一个小窗口。一般这个窗口要求你必须有回应，或者可以忽略掉，这就引出了关于对话框的两种分类：

  • 模态对话框 Modal Dialog
  • 非模态对话框 Modaless Dialog

这两种分类的区别就在于模态对话框需要你强制的回应它，而非模态不需要回应它。前面说到的AntD组件中把对话框叫做Model原因也在此，在AntD设计语言中仅仅把模态对话框叫做对话框。

0x01 什么场景下使用

那么什么时候使用对话框呢？在AntD官方描述上是这样说的：

需要用户处理事务，又不希望跳转页面以致打断工作流程时，可以使用 Modal 在当前页面正中打开一个浮层，承载相应的操作。

同样在Element上的描述是这样的：

Dialog 弹出一个对话框，适合需要定制性更大的场景。

常见的模态对话框都是这样的，一般出现在页面中，显示关键信息，这么做的原因是可以打断用户的信息流，使用户可以专注于模态对话框上任务。

而在Element上，还提出了另外一种模态对话框的样式，可以处理表单信息。在早期的互联网产品上，这种形式不少见，更有甚者还有在模态对话框上承载了大量的表单信息让用户去操作。

在我后来多次工作中也使用过类似的设计，初看觉得没有什么问题，感觉这种设计方式与在一个新的页面下填写表单的区别不大。但后来越发觉得这个设计方式不可取。原因有以下几点：

  1. 在需要大量填写表单的情况下，和本来页面不是同属于一个工作流，可以直接跳出。
  2. 在前端实现结构中，相对于新的页面填写方式过于复杂。
  3. 新页面方式不会打断原有的工作流。

0x02 我的工作经验总结

所以在我个人使用上来说：

  • 提示类的信息使用alert形式（非模态）的弹窗。
  • 提示类需要交互行为（确认行为，简单的信息填写，不超过三个字段）的使用模态的弹窗。
  • 表单填写尽量不使用弹窗形式。
